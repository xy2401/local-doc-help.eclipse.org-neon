Spawning a Subprocess

The spawn process is a sub-process that is started ad-hoc in the scope of an active process instance in runtime environment by users. These sub-processes cannot be specified at the time of workflow design. So a spawned sub-process behaves asynchronously with respect to execution at the time of:

  • Abort
  • Recovery
  • Process Hierarchy

It is possible to spawn all processes from all active models that are startable and you are authorized to start, including the most recent version of the same model.

Common Usage Pattern

The spawn subprocess feature helps you to specify, manage and monitor ad-hoc work in addition to structured business process execution. It allows teams to work on business cases collaboratively without having a predefined structure for this work. It helps to improve:

  • Operational efficiency
  • Auditing capabilities

The following organization scenario explains how Stardust can help you create ad-hoc process instances. Thus, supports unstructured workflow.

Assume that the organization has recently implemented a Scan client to scan documents into the Stardust Process Portal and start process instances. The Scanning implementation is generally regarded as a success but two issues have arisen that the organization has requested to be addressed with spawn subprocess feature in the Stardust Portal.

Business Case

This business case describes common usage pattern for the process instances that are spawned during the workflow processing. The spawning of a root process instance creates another sub-process of the same root process. The single document is available to all the spawned process instances.

Goal

This use case explains how the spawning of the process can streamline the operation.

Business Scenario 1

With approximately 5% of scans, the scan operator selects the incorrect process in the Scan UI and the document enters the workflow in the wrong process definition. When these errors are discovered these process instances are aborted and the documents are rescanned into the correct process definitions. This can be time-consuming and has resulted in multiple fines when the processing SLA's were missed. The organization has requested enhancements to allow operators working in the Stardust Portal to move these Process Instances (PIs) directly to the correct process definitions. Also, any data associated with the documents should be moved to the new process instances.

Business Scenario 2

The organization often receives work that contains multiple requests for a single client. Due to Auditing requirements, these documents must be scanned into Stardust as a single document and any additional process instances needed to work the client's requests should be started in the portal and referenced to this single document. The organization has requested enhancements that would allow operators in the Stardust to spawn a process instance for each of these requests and have each process instance reference the original document. Additionally, these tasks should have a linkage so that business units can track these requests as a single entity.

Example Usage

  • Create process definitions called Loans, Brokerage, HELOC and IRA
  • Scan a document into the Loans Process Definition using Stardust Scan application.
  • Log into Stardust Portal and locate the Loans process instance created in step 1.
  • Activate an Activity in the newly created Process Instance.
  • Click the Spawn Process icon to launch the Spawn Process dialog.
  • Use Spawn Process functionality to start a Brokerage and HELOC, process.
  • Review the newly started processes and confirm that the document/document data is linked to the source Loans process instance.
  • Review the process history of each newly started process and confirm that they are shown as sub-processes of the root process instance.
